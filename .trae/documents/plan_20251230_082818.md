# 邀约记录操作按钮显示逻辑优化方案

## 1. 核心问题分析

当前邀约记录操作按钮仅基于邀约状态显示，未考虑医生当前的实际状态（是否已成为专家），可能导致用户看到不必要的操作选项，影响用户体验。

## 2. 数据关联需求

为了实现合理的操作按钮显示逻辑，需要在邀约记录中关联医生的当前状态数据：

* **医生当前状态**：是否已成为专家

* **专家认证时间**：如果已成为专家，记录认证时间

* **专家认证方式**：通过邀约还是自行申请成为专家

## 3. 优化后的操作按钮显示逻辑

### 3.1 医生当前未成为专家的情况

| 邀约状态 | 操作按钮 |
| ---- | ---- |
| 待处理  | 取消邀约 |
| 已接受  | 无    |
| 已拒绝  | 重新邀约 |
| 已过期  | 重新邀约 |
| 已取消  | 重新邀约 |

### 3.2 医生当前已成为专家的情况

| 邀约状态 | 操作按钮 |
| ---- | ---- |
| 待处理  | 取消邀约 |
| 已接受  | 无    |
| 已拒绝  | 无    |
| 已过期  | 无    |
| 已取消  | 无    |

## 4. 邀约记录列表增强

### 4.1 新增医生当前状态标识

* 在专家信息列增加医生当前状态标识

* 示例：「张医生 (已成为专家)」

* 使用不同颜色区分：已成为专家（绿色）、未成为专家（灰色）

### 4.2 新增专家认证信息

* 在专家信息列增加专家认证方式和时间

* 示例：

  * 已通过邀约成为专家（2025-12-29）

  * 已通过自行申请成为专家（2025-12-30）

### 4.3 操作按钮显示规则

* 当医生已成为专家时，仅显示「取消邀约」（针对待处理状态）

* 当医生未成为专家时，根据邀约状态显示相应操作按钮

## 5. 具体实现建议

### 5.1 前端页面调整

1. 修改邀约记录列表的专家信息列，增加医生当前状态标识
2. 优化操作按钮的显示逻辑，结合医生当前状态
3. 在邀约记录中增加专家认证相关信息的展示

### 5.2 后端数据支持

1. 在邀约记录API中关联返回医生当前状态数据
2. 确保数据实时更新，反映医生最新状态
3. 提供医生状态变更的历史记录

### 5.3 交互体验优化

1. 鼠标悬停在医生状态标识上时，显示详细的专家认证信息
2. 对于已成为专家的医生，鼠标悬停在操作列时，显示提示信息：「该医生已成为专家，无需再次邀约」
3. 提供批量操作功能，支持批量查看医生当前状态

## 6. 预期效果

* 用户可以清晰了解每个邀约记录对应的医生当前状态

* 操作按钮的显示更加智能合理，避免无效操作选项

* 提供完整的专家认证信息，便于用户判断后续操作

* 提升邀约记录管理的效率和准确性

## 7. 技术实现优先级

1. 首先在前端页面增加医生当前状态标识
2. 然后优化操作按钮的显示逻辑
3. 最后完善专家认证信息的展示和交互
